## Slide 3 — 為什麼需要 CSI？

在進入技術細節之前，先確立我們為什麼需要 CSI。

RSSI 是最傳統的無線量測指標，但它只告訴你總接收功率，無法描述通道的頻率特性。RSSI 是整個頻帶的總接收功率，是一個純量值，單位 dBm。它告訴你「訊號有多強」，但不告訴你「通道長什麼樣子」——而且無法區分多路徑效應，也沒有頻率解析度和相位資訊。

CSI 則完全不同。CSI 是每個 OFDM 子載波的複數通道增益，包含振幅和相位。這個豐富的資訊讓 CSI 成為室內定位、手勢辨識、呼吸監測等研究的核心工具——這些應用用 RSSI 根本做不到。

今天的核心問題是：研究人員如何在實際硬體上取得 CSI？而 Windows 為什麼做不到？

## Slide 4 — 802.11 RF 信號路徑

這張圖應該是大家最熟悉的起點。從天線接收到 RF 信號後，經過 ADC 轉換成數位信號，再透過 OFDM 解調把各個子載波分離出來，最後解碼成 802.11 frame。

**關鍵點在這裡**：CSI 的計算發生在 OFDM 解調這個步驟。每個子載波在解調過程中，NIC 的 DSP 會計算出複數增益，這就是 CSI 的來源。這個資料此時還在 NIC 的韌體裡，尚未進入任何 OS 軟體層。

---

## Slide 5 — PHY / MAC 與 CSI

在座各位應該都知道 RSSI，但我還是簡單對比一下。RSSI 是整個頻帶的總接收功率，單位 dBm。

CSI 則是每個 OFDM 子載波的複數增益，包含振幅和相位。802.11n 的情況下，IWL5300 會報告 30 個子載波群組的 CSI。這個資訊豐富多了——你可以從 CSI 重建多路徑傳播特性，這就是為什麼 CSI 可以用於室內定位和手勢辨識，而 RSSI 不行。

CSI 的數學表示是頻域通道矩陣 H(f)，對每對 TX-RX 天線各有一個。

MAC 層的職責是 CSMA/CA、ACK、Frame 組裝，CSI 根本不在 MAC 協定的範圍內。PHY 解調完成後，CSI 儲存在 NIC 韌體暫存器，MAC 只收到已解調的 802.11 frame。這是今天整個故事的第一個截斷點。

## Slide 7（橋接引言）— CSI 要去哪？

現在進入 Windows 驅動架構。這是今天的核心問題：CSI 在 NIC 韌體計算完成後，要怎麼到達你的應用程式？

從 RF 工程師的視角來看，左邊是熟悉的領域：PHY 解調、NIC 韌體計算 CSI、儲存在暫存器。右邊是研究目標：你的程式需要 CSI 來做定位或感測。

中間那個「黑盒子」就是 Windows 的驅動堆疊。接下來我們**從最靠近硬體的元件開始，逐層往上**解析，找出 CSI 究竟在哪裡消失。

---

## Slide 8 — OEM.sys：網卡廠商 Miniport Driver

OEM.sys 是網卡廠商提供的 Miniport Driver，是整個 Windows 網路堆疊中距離硬體最近的軟體元件。

它透過 MMIO 或 PCI 介面直接讀寫網卡的硬體暫存器，可以控制 RF 參數像是頻道和 TX 功率，也負責管理 DMA 傳輸。

右邊的例子列出各廠商的驅動名稱。Intel 在 Linux 上叫 iwlwifi，但在 Windows 上叫 netwtw*.sys，星號代表版本號。

**這裡有一個關鍵觀察**：OEM.sys 理論上可以讀取 NIC 韌體計算的 CSI 資料，因為它有直接訪問硬體暫存器的能力。問題是它選擇不向上層暴露這個資料。

---

## Slide 9 — OEM.sys 的資料路徑

這張流程圖展示了一個 RX 封包從 NIC 韌體到應用層的路徑。

注意最左邊：NIC 韌體計算 CSI 後，把結果儲存在硬體暫存器。OEM.sys 透過 DMA 讀取 RX 封包緩衝區，但這個緩衝區裡只有 802.11 frame 的內容，**不含 CSI**。

CSI 從來沒有進入 NDIS_NET_BUFFER 這個標準封包結構。換句話說，CSI 在 OEM.sys 這一步就已經被排除在資料流之外了。

這就是 CSI 消失的根本原因：不是某個軟體元件「擋住」了 CSI，而是 CSI **從一開始就沒有被放進標準的資料傳輸路徑**。

---

## Slide 10 — nwifi.sys：連線管理層

nwifi.sys 是 Windows Native WiFi 驅動，職責是管理 802.11 連線狀態：認證（WPA2/WPA3）、漫遊、省電、掃描。

它向上暴露的介面包括 SSID、RSSI、連線狀態、頻道、連線速率，透過 Native WiFi API（wlanapi.dll）提供給應用程式。

但是——nwifi.sys **完全不暴露任何 PHY 層原始資料**：沒有 CSI、沒有 IQ 樣本、也沒有每天線獨立的 RSSI。這是設計目標，不是疏漏。

從 nwifi.sys 向上，PHY 層資訊已被永久丟棄。

---

## Slide 11 — NDIS 無 CSI 的根本原因

NDIS 的核心設計目標是跨廠商通用性，讓任何協定可以跑在任何網卡上。

OID 機制提供了 SSID、RSSI、掃描、統計等功能，但完全沒有任何 OID 定義 CSI 查詢。

為什麼無法標準化 CSI？因為不同廠商的 CSI 格式完全不同：Intel 5300 報告 30 個子載波群組，Atheros AR9380 報告 56 個子載波，格式完全不同。**NDIS 無法取這兩者的最大公約數**，所以根本沒有嘗試。

結論：通用性的代價就是丟棄廠商私有的 PHY 細節。

---

## Slide 12 — WDI：Windows 10+ 新驅動模型

Windows 10 引入了 WDI 作為新的 WiFi 驅動模型。WDI 把驅動分成兩部分：IHV 廠商負責的 User Mode 元件，和 WDI 框架的 Kernel Mode 元件。

這個架構的優點是：User Mode 的 crash 不會導致藍屏，大幅提升系統穩定性，也簡化了廠商的驅動開發。

但是——WDI 的 Task 和 Property 定義完全沒有包含 CSI 相關的介面。TaskScan、TaskConnect、TaskDisconnect、TaskSendRequest——全都是連線管理，沒有 CSI。

WDI 是架構上的改進，不是解決 CSI 取得問題的嘗試。即使升級到 WDI 模型，CSI 依然無法取得。

從最底層的 NIC 硬體開始，往上依序是 OEM.sys（韌體廠商驅動）、nwifi.sys（WiFi 連線管理）、NDIS、tcpip.sys、WinSock，最終到應用程式。

## Slide 14 — CSI 消失點分析

這是今天的**核心投影片**。讓我們總結 CSI 在哪裡消失。

橙色和紅色標注的兩個元件是關鍵：OEM.sys 和 nwifi.sys。

**OEM.sys** 有能力讀取 CSI（因為它可以直接訪問 NIC 硬體暫存器），但它不上報給 nwifi.sys。**nwifi.sys** 也從未向 OEM.sys 請求 CSI 資料。這個邊界就是 CSI 消失的地方。

再往上，ndis.sys 沒有 CSI 的 OID 定義，tcpip.sys 只看到 IP 封包，應用程式完全感知不到 CSI 的存在。

結論：**CSI 的消失是多層架構決策累積的結果**，每一層都有其合理性。

## Slide 15 — 為什麼 Linux 可以取得 CSI？

現在我們來看為什麼 Linux 可以取得 CSI。

Linux 的關鍵優勢是**開源**。iwlwifi、ath9k、brcmfmac 這些驅動都是開源的，研究人員可以直接修改驅動程式碼，甚至修改 NIC 韌體。debugfs 提供了一個核心介面，讓驅動可以暴露任意的除錯資料，包括 CSI。

右邊列出了 Windows 的根本差異：OEM.sys 是閉源二進位，不可修改；NDIS 介面固定，沒有擴充機制；韌體修改需要 WHQL 認證簽章。

這**不是 Windows 的 bug**，而是商業產品（Windows）vs 開源生態（Linux）的架構哲學差異。Linux 的可行性來自開源生態，不是 Windows 的設計缺陷。

## Slide 16 — 現代 CSI 工具生態（全部 Linux-only）

這是目前（2026 年）的現代 CSI 工具生態。

**PicoScenes** 是目前功能最完整的平台，支援 Intel AX200/AX210，可以取得 802.11ax 格式的 CSI，最寬支援 160 MHz。它有商業支援，適合需要穩定維護的研究團隊。

**FeitCSI** 是開源替代方案，功能相近，也支援 CSI 注入，可以用於主動量測。

這張表的「平台」欄請注意：**全部都是 Linux**。這個事實本身就是今天演講的核心論點之一——所有成熟的 CSI 工具都選擇 Linux，因為 Windows 的架構限制讓這件事在 Windows 上無法實現。

---

## Slide 17 — Windows 替代方案比較

四種方案快速比較：

- **Raw Packet Capture**（WinPcap/Npcap）：最簡單，但完全沒有 CSI，只能用於封包分析。
- **OEM 私有 IOCTL**（逆向 OEM.sys）：技術難度極高，CSI 取得不穩定，且每次驅動更新後可能失效。不建議。
- **雙系統 Linux**（推薦）：難度中等，取得完整 CSI，維護成本低，是最佳方案。
- **Linux Live USB**（FeitCSI）：最容易上手，完整 CSI，適合快速驗證。

結論：Intel AX200 + 雙系統 Linux + PicoScenes/FeitCSI 是今天能開始做 CSI 研究最低阻力的路徑。

## Slide 18 — 支援 CSI 的硬體選型

這張表列出目前（2026 年）有明確 CSI 工具支援的網卡。

如果你要買新硬體，**Intel AX200 M.2** 是最容易取得、工具支援最完整的選擇。它便宜（通常 500 到 1000 台幣），支援 802.11ax，PicoScenes 和 FeitCSI 都支援它。買 M.2 版本可以裝在筆電或桌機，或者用 M.2 轉 PCIe 轉接卡。

如果你有 Raspberry Pi，**BCM43455** 晶片已經內建，Nexmon CSI 直接支援，不需要額外購買硬體，非常適合做嵌入式 CSI 採集節點。

**Atheros AR9380** 在 OpenWRT 路由器上很常見，適合布署多點 CSI 量測環境。

請避免購買沒有在這份清單上的網卡，除非你願意自己移植驅動。

## Slide 19 — 完整架構總覽：CSI 的旅程

讓我們用這張總結圖回顧整個旅程。

從最下面的 PHY 硬體開始，OFDM 解調計算出 CSI，儲存在 NIC 韌體的暫存器中。NIC 韌體計算完成後，這個資訊就在這裡等待被讀取。

**OEM.sys** 是唯一有能力讀取這個資料的軟體元件，但它的設計目標不包含上報 CSI，所以它沒有這樣做。

往上的每一層，從 nwifi.sys 到 NDIS 到應用程式，都沒有任何機制可以取得 CSI。每一層都有合理的設計動機，但合在一起的結果就是 CSI 無法從應用層存取。

這不是一個單一決策造成的問題，而是多層架構設計累積的結果。

## Slide 20 — 給 RF 工程師的建議

基於今天的內容，給各位三點具體建議。希望能幫助大家避免走彎路，直接使用最有效的工具：

**第一點，選硬體先確認 CSI 工具支援。** Intel AX200/AX210 是目前最佳選擇，工具生態最完整，價格也合理。

**第二點，優先在 Linux 環境採集 CSI。** 可以用 PicoScenes（有商業支援）或 FeitCSI（開源）。雙系統是目前最務實的方案：Linux 採集 CSI，Windows 做後處理分析。

**第三點，避免這三個坑：**
- OEM 私有 IOCTL：每次驅動更新後可能失效，工程成本極高
- 虛擬機取 CSI：時序精度受 Hypervisor 影響，資料品質不穩定
- 沒有在支援清單上的網卡：沒有工具支援，移植難度很高

## 附錄：常見 Q&A

**Q：有沒有辦法在 Windows 取得近似 CSI 的資訊？**
A：有限度的。部分廠商的 Windows 驅動透過私有 IOCTL 暴露部分統計資訊，但格式未公開，且每次驅動更新後可能失效。不適合研究用途。

**Q：未來 Windows 有可能支援 CSI 嗎？**
A：可能性低。需要 Microsoft 主動在 NDIS/WDI 標準加入 CSI 介面，且所有 IHV 配合實作。目前沒有跡象顯示這個方向有任何進展。

**Q：WSL2 可以取 CSI 嗎？**
A：不行。WSL2 使用虛擬化的網路介面，無法存取真實的 NIC 硬體。需要原生 Linux 環境才能取得 CSI。

**Q：Intel AX200 在 Windows 上能取什麼資訊？**
A：透過 Windows 標準 API 只能取得 RSSI、SSID、頻道、連線速率等基本資訊。CSI 在 Windows 上無法存取。
